Skills and roles declare dependencies on other packages. The CLI resolves these recursively and installs them in topological order.

## Declaring Dependencies

**Skills** depend on other skills via a flat list of slugs:

```yaml
metadata:
  strawpot:
    dependencies:
      - security-baseline
      - git-workflow
```

**Roles** depend on both skills and roles via a structured object:

```yaml
metadata:
  strawpot:
    dependencies:
      skills:
        - git-workflow
        - code-review
      roles:
        - reviewer
```

The `roles` list supports `"*"` as a wildcard meaning "all available roles." This is useful for orchestrator roles that can delegate to any installed role. The wildcard is expanded at runtime by StrawPot and filtered out during install.

See [Frontmatter Reference](/strawhub/publishing/frontmatter) for the complete schema.

## Resolution Algorithm

When you install a role with dependencies:

1. Fetch the role from the registry
2. Read `metadata.strawpot.dependencies` from its frontmatter
3. For each dependency, recursively fetch and read its dependencies
4. Topological sort with cycle detection
5. Return install order (leaves first)

**Skill resolution** is done client-side by the CLI, recursively reading frontmatter.

**Role resolution** is done server-side via the `/api/v1/roles/:slug/resolve` endpoint, which returns the complete dependency tree.

## Project File (`strawpot.toml`)

Declare project dependencies in `strawpot.toml`:

```toml
[skills]
git-workflow = "*"
code-review = "==2.1.0"
security-baseline = "*"

[roles]
implementer = "*"
reviewer = "==2.0.0"
```

### Version Specifiers

Version specifiers are supported only in `strawpot.toml` — not in SKILL.md or ROLE.md frontmatter.

| Format | Meaning |
|--------|---------|
| `"*"` | Any version (install latest) |
| `"==X.Y.Z"` | Exact version |

Versions follow [semver](https://semver.org/) (`major.minor.patch`).

### Commands

```bash
# Install all dependencies from strawpot.toml
strawhub install

# Generate strawpot.toml from installed packages
strawhub init

# Update all and save new versions
strawhub update --all --save
```

## Directory Structure

### Local Scope (project-level)

```
./
├── strawpot.toml              # Project file (commit to VCS)
└── .strawpot/                 # Local package store
    ├── skills/
    │   ├── code-review/
    │   │   └── SKILL.md
    │   └── git-workflow/
    │       └── SKILL.md
    ├── roles/
    │   └── implementer/
    │       └── ROLE.md
    └── strawpot.lock          # Auto-generated lockfile (JSON)
```

### Global Scope

```
~/.strawpot/                   # Or STRAWPOT_HOME
├── skills/
├── roles/
└── strawpot.lock
```

### Scope Resolution

- **Install**: defaults to local. Use `--global` for global.
- **Resolve**: checks local first, then global. Local takes priority.
- **Project file**: local only. `--save` flags are incompatible with `--global`.
